微服务系统概述

为什么要学习微服务架构

单体应用架构存在的问题

一个归档包（例如war，jar）包含所有的功能的应用程序，通常成称为单体应用。而架构单体应用的方法论，就叫做单体应用架构
现如今呢，软件开发领域，绝大部分的应用还是采用的单体应用这种架构方式，那么有过开发经验的同学呢，想必对单体应用开发也有一定的了解。
单体应用的部署，测试等在项目初期还是比较好运行的。然而随着需求的不断增加，越来越多的人加入到开发团队，代码库也在飞快地膨胀。慢慢的
，单体应用就会变得非常臃肿，可维护性，扩展性，灵活性都要受到巨大的挑战，下面列举一些单体应用之间存在的问题

  1. 复杂性高：
    当一个单体应用扩展到百万行级别的时候，整个项目包含的模块非常多，彼此之间边界模糊，依赖关系不清晰，代码质量参差不齐，混乱的堆砌在一起，
    整个项目就变得非常的复杂，每次修改代码都非常的心惊胆战，可能就修改个简单的bug，就会带来10个其他的bug，甚至导致系统崩溃
  2. 技术债务：
    随着时间的推移，需求的变更，人员的更迭，会逐渐形成应用程序的技术债务，并且会越积越多。”不坏不修”，在软件开发中非常常见，在单体应用中
    这种思想更甚
  3. 部署频率低
    单体应用随着代码量的增多，在单体应用中，每次功能的变更和bug的修复，都会导致重新部署整个项目，你需要在经历一遍打包，部署。这就使得单体应用形目上线部署的
    频率低，更新时间长，出错概率比较高
  4. 可靠性差
    单体应用的某个BUG，例如死循环，OOM等，都可能使得整个系统崩溃
  5. 扩展能力受限
    单体应用的硬件资源只能根据整体应用扩展，没办法根据单独的业务模块进行差异化扩展
  6. 阻碍技术迭代重构
    单体应用往往使用统一的技术平台或者方案解决所有问题，团队中必须每个成员使用相同的开发语言和框架，当技术变更，引入新的框架和新技术的时候
    就会变得异常困难，切换成本无疑是非常高的

因此，综上所述，单体应用的架构模式已经很难满足互联网时代业务快速变化的需求。我们急需一种新的技术架构方式来解决这些问题。
所以，微服务架构，应运而生。

到底什么是微服务架构

就目前来看，微服务本身并没有一个准确的严格定义，每个人的理解都不尽相同，而我在诸多书中了解到，认为微服务架构，应该具备以下特点

  1. 每个微服务都可独立运行在自己的进程中
  2. 一系列独立运行的微服务共同构建起一整个系统
  3. 每个服务为独立的业务开发，一个微服务只关注某个特定的功能，例如订单管理，用户管理
  4. 微服务之间通过一些轻量的通信机制进行通信，例如 rest
  5. 每个服务，可以使用不同的技术栈，语言，数据存储技术
  6. 全自动化部署

这里能，可能有些人会搞混分布式和微服务，如果有时间博主会专门写一篇番外篇，来详细说明分布式和微服务的区别
这里先简单理解一下，微服务是架构设计方式，分布式是系统部署方式，两者的概念不同
一个微服务架构的应用，肯定是用分布式系统部署的，一个用分布式部署的应用不一定是微服务架构的，比如：集群部署
它是把相同应用复制到不同服务器上，但是逻辑功能上还是单体应用。

微服务架构的优点和缺点

优点：

  1. 易于开发和维护
    一个服务只专注于一个特定的业务功能，业务清晰，代码量较少
  2. 局部修改易于部署
    一般来说，每个服务都是单独部署，修改了某个服务，只需重新部署这个服务就可以了
  3. 技术栈不受限
    不同的服务，可以使用不同的技术栈，根据团队情况，敏捷度，灵活度更高
  4. 按需伸缩
    可以按需，对不同的服务分配不同的硬件资源，实现差异化。

当然这个世界没有十全十美的东西，解决一个问题，必将带来新的问题，微服务同样面临着很多的挑战于困难

缺点：

  1. 运维要求高
    这点，显而易见，更多的服务，意味着要付出更多的运维投入
  2. 分布式固有的复杂性
    对于一个分布式系统，系统容错，网络延迟(这点相信在将来的5G时代将不再成为问题)，分布式事务等
  3. 接口调整成本高
    微服务之间通过接口进行通信，如果要修改某个服务的API，可能所有使用该API的服务都需要调整

  4. 重复劳动
    很多服务都会做一些相同事情，这些功能没有达到分解为一个单独服务的程度，而每个服务可能又都需要开发这个功能，就会造成代码重复
    虽然可以通过共享库来解决这个问题，但共享库在多语言的环境下就行不通了

    微服务系统架构的设计原则

    就像数据库设计中的N范式一样，微服务也有一定的设计原则，这些原则将会指导我们更好的架构微服务

  5. 单一职责原则
    这个原则通俗的讲就是，你的每一个微服务都是一个单独的，能够独立运行的，有界限的服务个体

  6. 服务自治原则
    服务自治原则是说，每一个服务，应具备独立的业务能力，依赖和运行环境，应该与其他服务高度的解耦
  7. 轻量级通信机制
    服务与服务之间应该通过轻量级的通信机制进行交互，常用的协议有：REST，AMQP，STOMP等
  8. 微服务粒度
    微服务的每个服务的划分界限，要适度，不能一味的把服务做小，不同的业务复杂度不同，划分标准也不一样

    如何实现微服务系统架构

这里能，这个系列学习，博主主要学的是Spring Cloud 作为微服务的开发框架，Spring Cloud 为微服务架构提供了
一整套完整的解决方案。当然除了Spring Cloud之外还有其他的技术框架，如 Dubbo，Dropwizard，armada等

总述：

目录结构

如图下图：在Linux 系统中，所有的文件都在一个根 / 目录中，下面又很多文件分区，学习Linux文件，首先要明确一个概念：
在Linux中，一切皆文件，一切的东西都是文件的形式在存在着

每个文件分区的意义如下（不同的人的分布名称不尽相同）：

  1. bin：存放普通用户使用的命令/bin/ls，bin/date，或者可执行文件
  2. sbin：存放管理员使用的命令/sbin/service，或者可执行文件
  3. dev： 存放设备的文件
  4. root：root用户的home文件
  5. home: 存放普通用户的目录
  6. proc: 虚拟文件系统，反映出的是内核，进程信息或者实时状态。
  7. user: 系统文件，相当于windows下的c:\Windows
    /user/local  软件安装目录，相当于c:\Program
/user/bin   普通用户使用的应用程序
/user/sbin  管理员使用的应用程序
/usr/lib    库文件
/user/lib64  库文件
  8. boot: 存放的是 系统启动的相关文件
  9. etc: 配置文件 如：系统相关网络配置 /etc/sysconfig/network 应用相关配置 /etc/ssh/sshd_config…
  10. lib and lib64 :库文件
  11. tmp: 临时文件（全局可写：进程产生的临时文件）
  12. var: 存放一些变化文件，如数据库，日志，邮件等

剩下的其他文件目录不常用：
media 移动设备挂载点
mnt 手工挂载设备的挂载点
misc automount进程挂载

路径访问

这个知识点，主要需要掌握 查看目录结构，根据路径访问文件文件夹等，主要要记住一些常用命令
首先文件路径： 1.绝对路径
/bin/。。。 从 根目录 / 开始
2.想对路径
../表示上一层目录 ./表示当前目录
bin/。。。== ./bin/。。。 在当前目录下的bin文件

常用命令：

  1. ls + 路径 命令
    查看该路径下的 文件有哪些 路径为空，就是当前目录
  2. cd + 路径 命令
    有路径：访问 该路径的目录
    路径为空：则访问用户的 home 目录
    路径为 - ：cd - 返回上一层路径

  3. pwd 命令
    显示 当前目录路径

    增删改查

    增

    创建文件： touch + 文件名
    touch b1.txt
    创建文件夹: mkdir + 文件夹名
    mkdir mytest

    删

    rm + 参数 + 路径文件 删除该文件（该命令慎用）
    最好 是配合cd 进入到那个文件夹 再用 rm + 文件名删除
    rm 一些参数：
    -f 强制删除（默认）
    -i 删除前询问是否操作
    -r 递归删除

    改

    这里主要讲 mv 移动指令，通过mv指令可以实现 文件移动，重命名，剪切等操作
    命令格式：mv [-fiv] source destination
    移动：mv file1 /home/dir 讲file1 移动到 home/dir下
    移动并改名：mv file1 /home/dir/file2 讲file1 移动到 home/dir下并改名file2
    重命名： mv file1 file2 当亲文件夹下转移并改名
    mv 一些参数：
    -f :直接移动不询问（默认）
    -i :若目标文件已经存在，询问是否覆盖
    -u ：如果目标文件已经存在，源文件比较新，就覆盖

    cp 复制命令
    命令格式：cp [-adfilprsu] 源文件(source) 目标文件(destination)

    cp [option] source1 source2 source3 ...  directory

    参数说明：
    -a:是指archive的意思，也说是指复制所有的目录
    -d:若源文件为连接文件(link file)，则复制连接文件属性而非文件本身
    -f:强制(force)，若有重复或其它疑问时，不会询问用户，而强制复制
    -i:若目标文件(destination)已存在，在覆盖时会先询问是否真的操作
    -l:建立硬连接(hard link)的连接文件，而非复制文件本身
    -p:与文件的属性一起复制，而非使用默认属性
    -r:递归复制，用于目录的复制操作
    -s:复制成符号连接文件(symbolic link)，即“快捷方式”文件
    -u:若目标文件比源文件旧，更新目标文件
    如将/test1目录下的file1复制到/test3目录，并将文件名改为file2,可输入以下命令：
    cp /test1/file1 /test3/file2

    查

  4. find
    find /root test.txt 在/root目录下查找 test.txt
  5. grep -r
    (1)grep ‘test’ d*　　#显示所有以d开头的文件中包含 test的行
    (2)grep ‘test’ aa bb cc 　　 #显示在aa，bb，cc文件中包含test的行
    (3)grep ‘[a-z]{5}’ aa 　　#显示所有包含每行字符串至少有5个连续小写字符的字符串的行
    (4)grep magic /usr/src　　#显示/usr/src目录下的文件(不含子目录)包含magic的行
    (5)grep -r magic /usr/src　　#显示/usr/src目录下的文件(包含子目录)包含magic的行
    (6)grep -w pattern files ：只匹配整个单词，而不是字符串的一部分(如匹配’magic’，而不是’magical’)，
  6. which
    which grep 查询bin文件（添加过环境变量的）

安装Mariadb

yum安装

这里我们使用 yum 直接安装就好，非常简单 敲入命令即可
yum -y install mariadb mariadb-server

基础配置

1.完成安装后，首先就是启动Mariadb服务 和 设置开机自启
systemctl start mariadb //设置启动
systemctl enable mariadb //设置开机自启

2.然后进行Mariadb的相关配置
mysql_secure_installation //进行配置输入后，开头# 会变成如下
就会变成 Enter current password for root (enter for none):<– 这是在让你输入数据库密码，初次进入直接回车，没有密码

设置密码
Set root password? [Y/n] <– 这里输入Y
New password: <– 设置root用户的密码
Re-enter new password: <– 再输入一次你设置的密码

其他配置（后面就一路 Y 就完了）
Remove anonymous users? [Y/n] <– 是否删除匿名用户，回车
Disallow root login remotely? [Y/n] <–是否禁止root远程登录,回车,
Remove test database and access to it? [Y/n] <– 是否删除test数据库，回车
Reload privilege tables now? [Y/n] <– 是否重新加载权限表，回车

最后
mysql -uroot -p
登陆验证 大功告成

Navicat远程链接

Navicat 远程链接是通过网络进行链接，所以链接成功与否，与 Centos防火墙设置，Mariadb权限设置有关。
这里我们以 阿里云上服务器为例进行讲解。

设置Mariadb 用户权限

  1. 登陆Mariadb
    mysql -u root -p
  2. 切换到mysql数据库(这个数据库存的是整个Mariadb的信息，如用户信息)
    这里需要注意，在Mariadb中的命令是需要加 ; 的
    use mysql;
  3. 查看root用户信息
    select user,host from user;
    如果查看的结果如下图，说明你没有远程连接权限,host 表示这个用户只能在本机登陆
  4. 修改root用户权限，使其可以在任何机器上访问
    update user set host=’%’ where user =’root’;
    再 select user ,host from user; 查看
  5. 给root用户授权（所有权限）
    GRANT ALL PRIVILEGES ON . TO ‘root‘@’%’ IDENTIFIED BY ‘root’ WITH GRANT OPTION;
    然后刷新用户权限
    flush privileges;

    设置Centos7系统防火墙开放端口

    对Mariadb的用户权限设置完了之后，我们还需要对Centos7防火墙进行设置
  6. 查看防火墙运行状态
    firewall-cmd –state
    如果 状态是 runing 说明防火墙在运行
    如果 状态是 not runing 说明防火墙没在运行
    systemctl stop firewalld.service //关闭防火墙的命令
    systemctl start firewalld.service //启动防火墙的命令

虽说直接把防火墙关掉也能达到我们连接的目的，但这样很不安全所以建议打开

  1. 查看防火墙开放的端口号
    firewall-cmd –zone=public –list-ports //查看防火墙开放的端口号
    如果开启的端口中没有 3306端口，我们需要将其打开（数据库使用了这个端口）
  2. 开启端口
    firewall-cmd –zone=public –add-port=3306/tcp –permanent
    再 重新加载防火墙信息
    firewall-cmd –reload
    最后再次查看开启的端口有哪些
    firewall-cmd –zone=public –list-ports

最后开放阿里云平台的安全组（不是云平台，这步可以省略）

题意

gsy有一组砝码和天平秤。 每一个砝码的重量是1到10公斤之间的整数。 gsy想玩一下砝码和天平。她一个一个地把砝码放到天平的盘子上面。第一个砝码放在左盘，第二个砝码放在右盘，第三个砝码放在左盘，第四个砝码放在右盘，以此类推。gsy一共想放m 个砝码在盘上。

只是简单的把砝码放在盘子上面没意思，因此gsy制定了一些规则。首先，她不会将两个相同重量的砝码连续地放在秤上。也就是说，对任何一个 i来说，第 i个砝码和第 (i + 1)个砝码都是不同的(1 ≤ i < m)。其次，每次gsy在一个盘里放砝码的时候，都希望这个盘比另一个盘中。也就是说，相应盘里的砝码总和是严格大于另一个盘的。

现在你手头有所有gsy可用的砝码。可以假设对于每一种不同重量的砝码gsy有无数个可以用。你的任务是帮助郭思易放总数为 m 的砝码在天平上，或者告诉她办不到。

Input
第一行是一串由10个“0”或者“1”组成的字符串: 第 i个 (i ≥ 1) 数字为 “1” 表示gsy有重量为i 公斤的砝码, 否则该位置上的数字为 “0”。第二行有一个数字m (1 ≤ m ≤ 1000).

Output
第一行，如果可以按照规则将m 个砝码放在天平上，则打印”YES”, 否则打印 “NO”. 如果可以 m个砝码放在天平上，在下一行打印 m个整数表示砝码的重量，顺序按照你将它们放在天平上的顺序。

如果有多种解，打印一组即可。

Examples
Input
0000000101
3
Output
YES
8 10 8
Input
1000000000
2
Output
NO

题意

给你一个大小为n*n的只包含数字0、1的矩阵A.
我们称矩阵A有一个x压缩矩阵B,当且仅当n能被x整除,且对于任意的i,j (1<=i,j<=n).
总有A[i][j] = B[ceil(i/x)][ceil(j/x)].
其中ceil(x)表示x向上取整的结果,比如ceil(0.1)=ceil(0.3)=1.
比如矩阵A:
0011
0011
1100
1100
显然其有一个2压缩矩阵B:
01
10
现在已知矩阵A,请你求出一个最大的整数t,表示该矩阵拥有t压缩矩阵.
(注意每个矩阵显然有一个1压缩矩阵)
Input
输入的第一行包含一个整数n(4<=n<=5200),表示数据A的大小.
接下来n行,每行会有n/4个16进制数(保证n能被4整除).
(表示为0~9以及’A’~’F’,如0011会表示为3而1100会表示为C,你需将每个数字转化为4位二进制从而得到正确的A矩阵).
即所给的输入也是经过压缩的,你需要将其还原成原始A数组再解决本题。
Output
输出只有一个整数t.
Examples
【样例输入1】
4
3
3
C
C

【样例输出1】
2

【样例输入2】
4
0
0
0
0

【样例输出2】
4

思想

首先得到二进制的01矩阵，大体思路就是，遍历答案x ：1 ~ n ，再验证这个x压缩能否成立，成立，
则更新ans，不成立，向后遍历。
所以，重点就是如何进行验证，验证x压缩矩阵能否成立，这里可以利用二维前缀和，通过二维前缀和，何以快速
得到一个区间矩形的和num，如果num==0 或者 num==x*x 则成立，否则不成立。

解题报告-Floyd

发表于 2019-04-30 | 分类于 ACM算法 | 评论数：

题意：一张带权的无向图中有n个点,从1到n标号,一开始告诉你任意两点之间的最短距离是多少,dis[i][j]表示从i到j的最短距离。显然dis[i][j]=dis[j][i],保证一开始任意两个点之间都能互相到达.
现在又新加入了一些边,现在问你加入某一条边之后,所有点对之间的最短距离的和是多少。

思路

一道Floyd的水题，直接暴力加边，然后暴力遍历更新就好了。这里说下FLoyd思想
Floyd的整体思想就是，对于没一对点之间的距离，有两中情况，一种是用另一个点k作为中转，
dis[i][j] = dis[i][k]+dis[k][j],另一种就是 本身距离 dis[i][j] =dis[i][j];
Floyd模板：O(n^3)
