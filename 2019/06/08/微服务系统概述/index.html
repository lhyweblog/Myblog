微服务系统概述

发表于 2019-06-08 | 分类于 javaWeb | 评论数： | 阅读次数：

为什么要学习微服务架构

单体应用架构存在的问题

一个归档包（例如war，jar）包含所有的功能的应用程序，通常成称为单体应用。而架构单体应用的方法论，就叫做单体应用架构
现如今呢，软件开发领域，绝大部分的应用还是采用的单体应用这种架构方式，那么有过开发经验的同学呢，想必对单体应用开发也有一定的了解。
单体应用的部署，测试等在项目初期还是比较好运行的。然而随着需求的不断增加，越来越多的人加入到开发团队，代码库也在飞快地膨胀。慢慢的
，单体应用就会变得非常臃肿，可维护性，扩展性，灵活性都要受到巨大的挑战，下面列举一些单体应用之间存在的问题

  1. 复杂性高：
    当一个单体应用扩展到百万行级别的时候，整个项目包含的模块非常多，彼此之间边界模糊，依赖关系不清晰，代码质量参差不齐，混乱的堆砌在一起，
    整个项目就变得非常的复杂，每次修改代码都非常的心惊胆战，可能就修改个简单的bug，就会带来10个其他的bug，甚至导致系统崩溃
  2. 技术债务：
    随着时间的推移，需求的变更，人员的更迭，会逐渐形成应用程序的技术债务，并且会越积越多。”不坏不修”，在软件开发中非常常见，在单体应用中
    这种思想更甚
  3. 部署频率低
    单体应用随着代码量的增多，在单体应用中，每次功能的变更和bug的修复，都会导致重新部署整个项目，你需要在经历一遍打包，部署。这就使得单体应用形目上线部署的
    频率低，更新时间长，出错概率比较高
  4. 可靠性差
    单体应用的某个BUG，例如死循环，OOM等，都可能使得整个系统崩溃
  5. 扩展能力受限
    单体应用的硬件资源只能根据整体应用扩展，没办法根据单独的业务模块进行差异化扩展
  6. 阻碍技术迭代重构
    单体应用往往使用统一的技术平台或者方案解决所有问题，团队中必须每个成员使用相同的开发语言和框架，当技术变更，引入新的框架和新技术的时候
    就会变得异常困难，切换成本无疑是非常高的

因此，综上所述，单体应用的架构模式已经很难满足互联网时代业务快速变化的需求。我们急需一种新的技术架构方式来解决这些问题。
所以，微服务架构，应运而生。

到底什么是微服务架构

就目前来看，微服务本身并没有一个准确的严格定义，每个人的理解都不尽相同，而我在诸多书中了解到，认为微服务架构，应该具备以下特点

  1. 每个微服务都可独立运行在自己的进程中
  2. 一系列独立运行的微服务共同构建起一整个系统
  3. 每个服务为独立的业务开发，一个微服务只关注某个特定的功能，例如订单管理，用户管理
  4. 微服务之间通过一些轻量的通信机制进行通信，例如 rest
  5. 每个服务，可以使用不同的技术栈，语言，数据存储技术
  6. 全自动化部署

这里能，可能有些人会搞混分布式和微服务，如果有时间博主会专门写一篇番外篇，来详细说明分布式和微服务的区别
这里先简单理解一下，微服务是架构设计方式，分布式是系统部署方式，两者的概念不同
一个微服务架构的应用，肯定是用分布式系统部署的，一个用分布式部署的应用不一定是微服务架构的，比如：集群部署
它是把相同应用复制到不同服务器上，但是逻辑功能上还是单体应用。

微服务架构的优点和缺点

优点：

  1. 易于开发和维护
    一个服务只专注于一个特定的业务功能，业务清晰，代码量较少
  2. 局部修改易于部署
    一般来说，每个服务都是单独部署，修改了某个服务，只需重新部署这个服务就可以了
  3. 技术栈不受限
    不同的服务，可以使用不同的技术栈，根据团队情况，敏捷度，灵活度更高
  4. 按需伸缩
    可以按需，对不同的服务分配不同的硬件资源，实现差异化。

当然这个世界没有十全十美的东西，解决一个问题，必将带来新的问题，微服务同样面临着很多的挑战于困难

缺点：

  1. 运维要求高
    这点，显而易见，更多的服务，意味着要付出更多的运维投入
  2. 分布式固有的复杂性
    对于一个分布式系统，系统容错，网络延迟(这点相信在将来的5G时代将不再成为问题)，分布式事务等
  3. 接口调整成本高
    微服务之间通过接口进行通信，如果要修改某个服务的API，可能所有使用该API的服务都需要调整

  4. 重复劳动
    很多服务都会做一些相同事情，这些功能没有达到分解为一个单独服务的程度，而每个服务可能又都需要开发这个功能，就会造成代码重复
    虽然可以通过共享库来解决这个问题，但共享库在多语言的环境下就行不通了

    微服务系统架构的设计原则

    就像数据库设计中的N范式一样，微服务也有一定的设计原则，这些原则将会指导我们更好的架构微服务

  5. 单一职责原则
    这个原则通俗的讲就是，你的每一个微服务都是一个单独的，能够独立运行的，有界限的服务个体

  6. 服务自治原则
    服务自治原则是说，每一个服务，应具备独立的业务能力，依赖和运行环境，应该与其他服务高度的解耦
  7. 轻量级通信机制
    服务与服务之间应该通过轻量级的通信机制进行交互，常用的协议有：REST，AMQP，STOMP等
  8. 微服务粒度
    微服务的每个服务的划分界限，要适度，不能一味的把服务做小，不同的业务复杂度不同，划分标准也不一样
